# Architecture

## Overview

Agent Link has a unified entry point (`src/main.ts`) that supports four modes:

```
agent-link                           → standalone (local node + HTTP)
agent-link --accept-nodes            → local node + accept remote nodes
agent-link --accept-nodes --no-local → pure router (no local SDK)
agent-link --connect-to <url>        → node only, connect to remote panel
```

```
Browser <--HTTP/SSE--> Server
                        ├── Local dispatch (sessions, vscode)
                        └── Remote nodes via WebSocket
                              ├── Node A (NAT)
                              └── Node B (NAT)
```

## Running

### Standalone

```sh
bun run dev                   # port 3456
agent-link --port 8080        # custom port
```

### Multi-node

```sh
# 1. Start server accepting nodes
agent-link --accept-nodes --port 3457

# 2. Connect node(s)
agent-link --connect-to http://panel:3457 --name my-node
# First connect: pending until approved
# Reconnect: auto-approved (machineId persisted)
```

### Pure router (no local SDK)

```sh
agent-link --accept-nodes --no-local --port 3457
# All requests forwarded to remote nodes
```

## Machine ID

Each instance has a deterministic machine ID: `hostname-xxxxxxxx` (8 hex chars from SHA-256 of `/etc/machine-id`, fallback to hostname+arch+homedir).

- Used as both nodeId and registration key
- Stable across restarts, readable
- Generated by `src/identity.ts`

## Request Routing

```
HTTP Request → routes.ts → Router
                             ├── isLocal(nodeId)? → dispatch.ts → sessions/vscode
                             └── remote?          → requestNode() → WS → Node
```

- `dispatch.ts`: single switch mapping action names to `sessions.*`/`vscode.*` calls
- Used by both Router (local) and Node connector (remote request handling)
- `router.ts`: checks nodeId, dispatches locally or via `RemoteProvider`
- `routes.ts`: all API routes written once, delegates to Router

## Connection Flow

1. Node starts with `--connect-to <url>`, generates machineId from `identity.ts`
2. Connects WebSocket to `<panel>/ws/node`, sends `register { machineId, label }`
3. Panel stores record in `~/.agent-link/nodes.json` keyed by machineId
4. New node: `approved: false`, Panel replies `pending`
5. Admin approves via `POST /api/nodes/:nodeId/approve`
6. Node sends `heartbeat { activeSessionIds, vscodeServers }` every 10s
7. Panel pings every 30s; on disconnect, Node auto-reconnects (backoff 1s→60s)

## WS Protocol

### Node → Panel

| Message | Fields | Purpose |
|---------|--------|---------|
| `register` | machineId, label | Initial registration |
| `heartbeat` | activeSessionIds, vscodeServers | Status update |
| `event` | sessionId, event | Forward SDK events |
| `response` | requestId, data | Reply to request |
| `error` | requestId, error | Error reply |
| `tunnel:*` | tunnelId, ... | VSCode HTTP/WS tunnel |

### Panel → Node

| Message | Fields | Purpose |
|---------|--------|---------|
| `registered` | nodeId | Auth success |
| `pending` | | Awaiting approval |
| `request` | requestId, action, params | Forward API call |
| `ping` | | Keep-alive |
| `tunnel:*` | tunnelId, ... | VSCode HTTP/WS tunnel |

## Request Actions

| Action | Params | Maps to |
|--------|--------|---------|
| `query` | prompt, cwd, model, sessionId? | `POST /api/query` |
| `interrupt` | sessionId | `POST /api/interrupt/:id` |
| `setModel` | sessionId, model | `POST /api/model/:id` |
| `listSessions` | cwd?, limit?, offset? | `GET /api/sessions` |
| `getSessionInfo` | sessionId, cwd? | `GET /api/sessions/:id` |
| `getSessionMessages` | sessionId, cwd?, limit?, offset? | `GET /api/sessions/:id/messages` |
| `listVscodeVersions` | | `GET /api/vscode/versions` |
| `startVscodeServer` | cwd, commit | `POST /api/vscode/start` |
| `stopVscodeServer` | cwd | `POST /api/vscode/stop` |
| `getInstallCommand` | version? | `GET /api/vscode/install-command` |

## VSCode Tunnel

VSCode traffic goes through the Node's management WebSocket.

**URL**: `/vscode/<nodeId>/<encoded-cwd>/...` (multi-node) or `/vscode/<encoded-cwd>/...` (standalone)

**HTTP**: Panel sends `tunnel:request` → Node fetches local VSCode → `tunnel:response` → Panel rewrites `remoteAuthority` → browser.

**WebSocket**: Browser ↔ Panel ↔ `tunnel:ws-data` ↔ Node ↔ local VSCode WS.

## File Structure

```
src/
  main.ts           -- Unified entry point, CLI args, Bun.serve + WS handlers
  identity.ts       -- Machine ID generation (hostname-hash)
  dispatch.ts       -- action→function mapping (shared by router + connector)
  router.ts         -- Route by nodeId: local dispatch or remote forward
  routes.ts         -- All Hono API routes (written once)
  sessions.ts       -- Session lifecycle (Claude SDK wrapper)
  managed.ts        -- Managed sessions/folders CRUD (JSON file store)
  store.ts          -- JSON file persistence with mtime cache
  claude-sdk.ts     -- SDK abstraction layer
  vscode.ts         -- VSCode serve-web lifecycle
  assets.ts         -- Static asset imports for compiled binary
  protocol.ts       -- WS message type definitions
  panel/
    nodes.ts        -- Remote node pool, approve, request routing, event relay
    tunnel.ts       -- HTTP/WS tunnel manager
  node/
    connector.ts    -- WS client, auto-reconnect, heartbeat, event forwarding
    tunnel.ts       -- Local VSCode fetch/WS for tunnel messages
```
